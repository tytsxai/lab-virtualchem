# VirtualChemLab：注释与文档完善审查报告（维护安全）

本报告针对当前仓库进行一次“维护安全”视角的注释与文档审查，目标是：
让**不熟悉项目背景**的维护者，仅通过代码与文档就能正确理解设计意图、接口契约与高风险点，降低误改概率。

> 范围说明：仓库 `src/` 下 Python 文件约 384 个，无法逐行穷举。本次审查采用“入口链路 + 关键模块 + 对外接口 + 配置安全”抽样策略，并优先修复**会导致理解偏差**的文档漂移点。

## 3. 接口与协议说明（REST/Admin API）

### 3.1 总体现状（结论）

- ✅ 对外 REST/Admin API 的“事实来源”已集中到 `docs/API.md`（并对齐 `src/api/server.py`、`src/api/admin_api.py`）。
- ⚠️ `docs/API_REFERENCE.md` 存在与当前实现不一致的函数名/返回值描述（会导致维护者按文档调用时报错或误解返回结构）。
- ⚠️ `src/api/server.py` 的 `/api/docs`（OpenAPI JSON）为“轻量自描述”，但其中部分 schema 与真实响应结构不一致，会误导自动化客户端生成/集成方。

### 3.2 已处理项

- 已更新 `docs/API_REFERENCE.md`：
  - `TemplateEngine.list_experiments()` → `TemplateEngine.list_available_experiments()`
  - `ExperimentController.finish_experiment()` → `ExperimentController.complete_experiment()`
  - `submit_step()` 返回值由 tuple 改为 `StepResult`（并说明兼容 tuple 解包）
- 已更新 `src/api/server.py` 的 OpenAPI JSON（`/api/docs`）中部分 endpoint 的响应 schema，使其更贴近实际响应结构（尤其是 `/api/experiments`、`/api/records`、`/api/reports/generate`）。
- 已在 `docs/API.md` 补充说明：`/api/reports/generate` 返回的 `url=/reports/...` 当前是“相对路径提示”，stdlib `HTTPServer` 默认不会托管 `reports/` 静态文件，避免集成方误以为可直接 GET 访问。

### 3.3 建议（后续）

- 若计划提供“可被第三方稳定集成”的 API：
  - 建议统一成功响应 envelope（例如统一 `success/data/trace_id` 或统一 `record` 顶层），减少客户端分支逻辑。
  - 建议把 OpenAPI 生成改为“从路由/模型集中定义”，避免人工维护的 schema 漂移。

## 5. 易误改与高风险点（魔法值/边界条件/不可简化逻辑）

### 5.1 已识别的关键高风险点（建议保持醒目标注）

- 启动与安全：
  - `src/core/startup_preflight.py`：生产环境弱密钥 fail-fast（>=32）
  - `src/core/config_loader.py`：配置优先级与“不可写目录/打包环境”重定向逻辑
- API 暴露面：
  - `src/api/server.py`：默认回环绑定 + CORS 本机默认 + 认证 fail-closed
- 存储一致性：
  - `src/storage/json_store.py`：采用“临时文件 + fsync + rename”的原子写策略（避免断电/崩溃破损）
- 兼容/历史处理：
  - `src/security/data_protection.py`：兼容旧 AES-CBC 解密路径（不具备完整性校验），属于安全敏感逻辑

建议维护者对上述文件的变更视为“安全变更/需要同步更新文档”的类别（参见 `docs/MAINTENANCE_SAFETY.md`）。
